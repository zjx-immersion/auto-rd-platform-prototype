### 二、第6章核心流程增加（需求2）

#### 新增 6.4 Sprint执行与交付流程 ✅

**设计说明**: Sprint执行与交付流程承接PI Planning，通过持续计划、每日协同、进度跟踪、Review和Retrospective实现2周迭代的目标交付。

**流程关键步骤**:
```
1. Sprint Planning - 承接PI Planning，生成Sprint Backlog
2. Sprint启动会 - 目标宣贯，任务分配
3. 迭代循环（2周）:
   - Daily Standup (每日15分钟)
   - Task执行 (开发测试)
   - 进度跟踪 (燃尽图更新)
   - 协同解决 (阻碍依赖处理)
4. Sprint Review - Demo演示，DoD检查
5. Sprint Retrospective - 回顾改进
6. 同步到PI - PI看板更新，进度报告
7. 判断PI是否结束，否则继续下个Sprint
```

**关键活动表**:
| 阶段 | 活动 | 频率 | 关键输出 | 检查点 |
|------|------|------|---------|--------|
| Planning | Sprint Planning | Sprint开始 | Sprint Backlog | 承诺明确、容量合理 |
| 执行 | Daily Standup | 每日 | 问题清单 | 阻碍及时暴露 |
| 执行 | Task执行 | 持续 | 增量代码 | WIP限制、质量门禁 |
| Review | Sprint Review | Sprint结束 | 完成增量 | DoD检查通过 |
| 改进 | Retrospective | Sprint结束 | 改进行动 | 持续改进PDCA |

#### 新增 6.5 特性需求评审流程 ✅

**设计说明**: 特性需求评审流程包含PRD评审（单个Feature评审）和SSTS批量评审（多个SSTS统一评审），确保需求质量和团队共识。

**流程分支**:
```
分支1: PRD评审
1. PRD准备 - FO编写PRD，完整性检查
2. 评审准备 - 发起评审，邀请SE/SO/DL/QA
3. PRD评审 - 需求宣讲，验证完整性/可行性
4. 评审结果判断:
   - 不通过 → PRD修订 → 重新评审
   - 通过 → PRD发布 → 触发SSTS拆解

分支2: SSTS批量评审
1. SSTS拆解 - SE从PRD拆解SSTS（功能/技术/测试型）
2. 评审准备 - 批量打包，邀请SO/DL/QA
3. 批量评审 - 逐条评审，批注通过/问题/退回
4. 问题处理 - 修订有问题的SSTS，重审仅修订项
5. SSTS发布 - 批量发布，触发MR拆解
```

**评审检查项**:
| 评审类型 | 检查项 | 评审角色 | 通过标准 |
|---------|--------|---------|---------|
| PRD评审 | 需求完整性 | SE/SO/DL/QA | 功能描述清晰完整 |
| PRD评审 | 技术可行性 | SE/DL | 技术方案可实现 |
| SSTS批量评审 | 覆盖完整性 | SO/DL | 覆盖PRD全部条目 |
| SSTS批量评审 | 拆解合理性 | SO/DL | 粒度适中、可实现 |

#### 新增 6.6 产品需求版本管理与发布流程 ✅

**设计说明**: 产品需求版本管理流程管理产品下Feature和MR的版本定义、内容规划、评审发布、变更控制全生命周期。

**流程关键步骤**:
```
1. 版本创建 - PO根据产品路线图创建版本
2. Feature规划 - 选择Epic拆解Feature，分配到版本
3. Feature评审 - PRD评审，确认功能范围
4. SSTS拆解 - SE批量拆解SSTS，批量评审
5. MR分配 - SO将SSTS分配到Module，拆解MR
6. 版本锁定 - PO最终确认，锁定Feature/SSTS/MR
7. 版本发布 - 状态设为已发布，通知项目团队
8. 版本变更（如需要）:
   - 变更申请 → 影响分析 → 变更评审 → CCB决策
   - 批准 → 版本更新 → 变更通知
   - 拒绝 → 回到执行
9. 版本完成 → 版本关闭归档
```

**版本管理规则**:
| 阶段 | 可变更度 | 批准权限 |
|------|---------|---------|
| 创建/规划 | 高 | PO |
| 锁定 | 低 | PO+CCB |
| 执行中-小变更 | 中 | TPM |
| 执行中-大变更 | 低 | CCB |

#### 新增 6.7 项目基线管理流程 ✅

**设计说明**: 项目基线管理流程定义车型节点基线的设定、锁定、变更控制，基线关联产品版本、需求版本、代码版本，确保交付质量可控。

**流程关键步骤**:
```
1. 车型节点定义 - PM定义SOP/预量产/试制节点
2. 基线规划 - 节点与基线绑定，定义范围
3. 产品版本选择 - PO选择版本纳入基线
4. PI规划对齐 - PI计划对齐基线，检查交付能力
5. 基线评审 - 管理团队评审，确认范围/计划/风险
6. 基线锁定 - 锁定需求/代码/文档配置
7. 基线发布 - 正式发布，全员同步
8. 基线执行:
   - 无变更 → 基线验证 → 节点交付
   - 需要变更:
     a. 变更申请 (CCR)
     b. 影响分析 (节点/质量/成本)
     c. CCB评审
     d. 批准 → 基线更新 → 产品版本更新 → 计划调整 → 变更通知
     e. 拒绝 → 继续执行
9. 基线验证 → 节点交付 → 基线关闭归档
```

**基线类型与控制级别**:
| 基线类型 | 内容 | 锁定时机 | 变更控制级别 |
|---------|------|---------|------------|
| 需求基线 | Feature/SSTS/MR | 基线锁定前 | CCB审批 |
| 代码基线 | 源代码/配置 | 集成测试前 | CCB审批 |
| 节点基线 | 完整交付物 | 车型节点前 | 高层审批 |

#### 新增 6.8 需求全生命周期变更管理流程 ✅

**设计说明**: 需求全生命周期变更管理流程覆盖从Epic接受到需求池、PRD/SSTS拆解、MR细化、计划执行、迭代开发全过程的需求变更控制。

**流程分阶段控制**:

**1. Epic阶段变更**:
```
触发: 市场/客户反馈
位置: 需求池
权限: 高（可自由修改）
流程: Epic变更 → 价值评估 → 产品委员会决策 → Epic更新
```

**2. Feature阶段变更（PRD已拆解）**:
```
触发: 技术/市场变化
影响: 影响SSTS/MR/Task
流程:
  Feature变更 → 影响分析（识别SSTS/MR/Task）→ 变更评审
  → 判断: 纳入当前版本 or 延期到下版本
  → 纳入 → 版本变更CCB → 批准 → 同步
  → 延期 → 加入下版本 → 通知
```

**3. SSTS阶段变更（已拆解SSTS）**:
```
触发: 技术方案调整
影响: 影响MR/Task
流程:
  SSTS变更 → SSTS影响分析（识别受影响模块）
  → 团队评估（工作量、Sprint影响）
  → 变更批准（TPM）→ 计划/资源调整 → 同步
```

**4. MR阶段变更（开发前/中）**:
```
触发: 实现细节调整
影响: 影响Task
流程:
  MR变更 → 团队内评估
  → 判断影响范围:
     - Sprint内 → 团队内消化 → 同步
     - 跨Sprint/团队 → 上报TPM → 按SSTS变更流程处理
```

**5. Task阶段变更（开发中）**:
```
触发: 实现问题
影响: 团队内部
流程:
  Task变更 → 开发者处理（技术方案调整）
  → 判断: 需上报?
     - 不需要 → 团队内消化（Daily同步）→ 继续开发
     - 需要上报 → 按MR变更流程处理
```

**需求变更控制矩阵**:
| 需求阶段 | 变更触发 | 评估角色 | 批准权限 | 影响范围 | 控制级别 |
|---------|---------|---------|---------|---------|---------|
| Epic | 市场/战略 | PO | 产品委员会 | 全局 | 高 - 严格控制 |
| Feature(PRD已拆) | 技术/市场 | FO+SE+TPM | CCB | 版本/PI | 中高 - CCB审批 |
| SSTS(已评审) | 技术方案 | SE+SO+TPM | TPM | PI/Sprint | 中 - TPM审批 |
| MR(开发前) | 实现细节 | DL+DEV | DL/TPM | Sprint/Team | 中低 - DL审批 |
| Task(开发中) | 实现问题 | DEV | DL | Team内部 | 低 - 团队内消化 |

**变更管理5原则**:
1. **阶段越早，变更成本越低** - 鼓励早期发现问题
2. **权限分层** - 根据影响范围确定审批权限
3. **影响分析必须** - 所有变更需先影响分析
4. **CCB机制** - 重大变更需CCB评审
5. **全程追溯** - 变更记录完整可追溯
