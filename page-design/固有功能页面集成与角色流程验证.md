## 📖 文档说明

### 文档定位

本文档基于**固有功能导航（C0-C7能力域）**为主线，将平台所有页面按照功能流程和数据流集成，从9个用户角色的动态使用视角，验证：

1. ✅ 页面设计是否完整
2. ✅ 功能流程是否顺畅
3. ✅ 页面跳转是否合理
4. ✅ 数据流转是否正确
5. ✅ 角色协同是否高效

#### 数据流转

```
RequirementPool (选择需求)
    ↓ GET /api/requirements/pool
[需求池数据]
    ↓ 选中3个需求
EpicCreate (创建Epic)
    ↓ POST /api/epics
[Epic数据库]
    ↓ Epic ID生成
EpicDetail (查看Epic)
    ↓ 点击"发起评审"
EpicReview (评审流程)
    ↓ POST /api/epics/:id/review
[评审记录表]
    ↓ 评审通过
FeatureDecompose (拆解Feature)
    ↓ POST /api/features/batch
[Feature数据库 + Epic-Feature关联表]
    ↓
FeatureList (展示Feature列表)
    ← GET /api/features?epicId=EPIC-001
```

## 数据流转验证

### 核心数据实体流转图

```
┌──────────────────────────────────────────────────────────────────┐
│                    数据流转全景图                                 │
└──────────────────────────────────────────────────────────────────┘

需求层数据流:
───────────────────────────────────────────────────────────────────
Requirement (需求池)
    ↓ 创建
Epic (业务需求)
    ├─ 属性: ID, 标题, 描述, 优先级, MoSCoW, 业务价值
    ├─ 关联: Project (N:1), Version (N:1)
    └─ 状态流转: 待评审 → 已评审 → 已分配 → 开发中 → 已完成
    ↓ 拆解
Feature (特性)
    ├─ 属性: ID, 标题, PRD, 估算SP, 负责人
    ├─ 关联: Epic (N:1), PI (N:1), Team (N:M)
    └─ 状态流转: 待PRD → 已评审 → 已分配 → 开发中 → 测试中 → 已完成
    ↓ 拆解
SSTS (规格)
    ├─ 类型: 功能SSTS (FO) / 技术SSTS (SE)
    ├─ 属性: ID, 类型, 规格内容, 验证标准
    ├─ 关联: Feature (N:1)
    └─ 状态流转: 草稿 → 已评审 → 已分配 → 开发中 → 已验证
    ↓ 拆解
MR (Module Requirement)
    ├─ 属性: ID, Module, Team, 接口定义, 估算SP
    ├─ 关联: SSTS (N:M), Module (N:1), Team (N:1), Asset (N:1可选)
    └─ 状态流转: 待分配 → 已就绪 → Sprint中 → 开发中 → 已完成
    ↓ 拆解
Task (任务)
    ├─ 属性: ID, 类型, 描述, 估算小时, 分配人
    ├─ 关联: MR (N:1), Sprint (N:1), DEV (N:1)
    └─ 状态流转: 待办 → 进行中 → 评审中 → 已完成

资产层数据流:
───────────────────────────────────────────────────────────────────
Product Asset (产品资产)
    ↓ 包含
Feature Asset (特性资产)
    ├─ 属性: ID, 名称, 版本, 成熟度, 复用次数
    ├─ 关联: Product (N:1)
    └─ 成熟度: L1 → L2 → L3 → L4 → L5
    ↓ 包含
Module Asset (模块资产)
    ├─ 属性: ID, 名称, 版本, Team绑定, 技术栈, 性能指标
    ├─ 关联: Feature Asset (N:1), Team (1:1绑定)
    └─ 复用: MR关联Asset → 适配工作 → 集成验证

规划层数据流:
───────────────────────────────────────────────────────────────────
Project (项目)
    ↓ 包含
PI (Program Increment)
    ├─ 属性: ID, 名称, 开始/结束时间, 目标, 总容量
    ├─ 关联: Project (N:1), Feature (N:M), Team (N:M)
    └─ 状态: Planning → Active → Review → Closed
    ↓ 包含
Sprint (冲刺)
    ├─ 属性: ID, 名称, Team, 容量, 速率, 承诺
    ├─ 关联: PI (N:1), Team (N:1), MR (N:M), Task (1:N)
    └─ 状态: Planning → Active → Review → Closed

测试层数据流:
───────────────────────────────────────────────────────────────────
Test Plan (测试计划)
    ↓ 包含
Test Case (测试用例)
    ├─ 属性: ID, 标题, 前置条件, 测试步骤, 预期结果
    ├─ 关联: Feature (N:1), SSTS (N:M), MR (N:M)
    ├─ 来源: 验收标准 (Given-When-Then)
    └─ 状态: 草稿 → 待执行 → 执行中 → 通过/失败
    ↓ 执行产生
Defect (缺陷)
    ├─ 属性: ID, 标题, 严重程度, 优先级, 状态
    ├─ 关联: Test Case (N:1), Task (N:1), Sprint (N:1)
    └─ 状态流转: New → Assigned → In Progress → Fixed → Verified → Closed

发布层数据流:
───────────────────────────────────────────────────────────────────
Version (版本)
    ↓ 包含
Release Plan (发布计划)
    ├─ 属性: ID, 版本号, 发布时间, 发布内容
    ├─ 关联: PI (N:M), Feature (N:M), Sprint (N:M)
    └─ 状态: Planning → Testing → Staging → Production
    ↓ 执行
Deployment (部署)
    ├─ 属性: ID, 环境, 部署时间, 部署状态
    ├─ 关联: Release Plan (N:1), Environment (N:1)
    └─ 状态: Pending → Running → Success → Failed → Rollback

度量层数据流:
───────────────────────────────────────────────────────────────────
所有实体 → 度量数据采集
    ↓ 聚合计算
Metrics (度量指标)
    ├─ 需求度量: Epic完成率, Feature流转效率, 前置时间
    ├─ 开发效能: Sprint速率, 代码质量, 提交频率
    ├─ 质量度量: 缺陷密度, 测试覆盖率, 逃逸率
    ├─ 交付度量: 交付周期, 发布频率, 回滚率
    └─ 团队效能: 产能利用率, 流动效率, 预测准确性
    ↓ 展示
Dashboard / Report (仪表板/报表)
```
