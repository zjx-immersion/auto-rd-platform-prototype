# C2-F01: SSTS拆解

> **功能编号**: C2-F01  
> **功能名称**: SSTS拆解 (System/Subsystem Technical Specification)  
> **所属模块**: C2-方案设计  
> **所属流程**: P2-特性设计流程 (步骤3)  
> **优先级**: P0 (核心功能)  
> **设计日期**: 2025-01-15

---

## 一、功能概述

### 1.1 功能定位

SSTS拆解是特性设计流程（P2）的第三步，负责将PRD拆解为系统/子系统技术规格说明书。

### 1.2 核心价值

- ✅ 将产品需求转化为系统技术规格
- ✅ 定义系统架构和子系统划分
- ✅ 明确系统间接口和交互
- ✅ 支撑后续MR拆解
- ✅ 符合ASPICE SYS.2标准

### 1.3 需求流转

```
Epic → Feature → PRD → SSTS → MR
                 ↑      ↑
              P2步骤2  P2步骤3 (当前)
```

### 1.4 用户角色

- **主要用户**: 系统工程师、架构师
- **次要用户**: 技术负责人
- **评审用户**: 架构委员会、技术专家

## 三、SSTS模板结构

### 3.1 SSTS标准章节

| 章节 | 是否必填 | 说明 |
|-----|---------|------|
| 1. 系统描述 | 是 | 子系统功能描述 |
| 2. 功能需求映射 | 是 | 映射到PRD的功能需求 |
| 3. 技术规格 | 是 | 硬件、软件、算法规格 |
| 4. 接口定义 | 是 | 输入输出接口定义 |
| 5. 性能指标 | 是 | 可量化的性能指标 |
| 6. 安全要求 | 是 | 安全等级、故障处理 |

### 3.2 SSTS编号规则

```
SSTS-XXX: 系统/子系统技术规格
  ├─ SSTS-001: 第一个子系统
  ├─ SSTS-002: 第二个子系统
  └─ ...
```

## 四、字段说明

### 4.1 SSTS基本信息

| 字段名称 | 字段类型 | 是否必填 | 说明 |
|---------|---------|---------|------|
| ssts_id | String | 自动生成 | SSTS唯一标识 |
| ssts_name | String | 是 | 子系统名称 |
| prd_id | String | 是 | 关联的PRD ID |
| system_desc | Text | 是 | 系统描述 |

### 4.2 技术规格

| 字段名称 | 字段类型 | 是否必填 | 说明 |
|---------|---------|---------|------|
| hardware_spec | Text | 是 | 硬件规格 |
| software_spec | Text | 是 | 软件规格 |
| algorithm_spec | Text | 否 | 算法规格 |
| performance_spec | Text | 是 | 性能指标 |

### 4.3 接口定义

| 字段名称 | 字段类型 | 是否必填 | 说明 |
|---------|---------|---------|------|
| input_interface | Text | 是 | 输入接口 |
| output_interface | Text | 是 | 输出接口 |
| protocol | String | 是 | 接口协议 |

## 五、业务规则

### 5.1 SSTS拆解规则

1. **一个PRD拆解为多个SSTS**
2. **所有功能需求必须被SSTS覆盖**
3. **所有非功能需求必须被SSTS覆盖**
4. **SSTS之间的接口必须明确定义**
5. **技术规格必须可实现**

### 5.2 需求覆盖度检查

- 功能需求覆盖度 = 已覆盖FR数 / 总FR数
- 非功能需求覆盖度 = 已覆盖NFR数 / 总NFR数
- 要求: 覆盖度 = 100%

### 5.3 SSTS评审规则

1. **草稿状态 (Draft)**: 可编辑
2. **评审中 (Review)**: 只读，等待评审
3. **已批准 (Approved)**: 只读，可进入P3流程
