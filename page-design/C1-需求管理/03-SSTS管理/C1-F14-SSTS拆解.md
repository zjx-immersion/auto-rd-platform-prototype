# C1-F14 SSTS拆解

> **功能编号**: C1-F14  
> **功能名称**: SSTS拆解（系统/技术/安全需求拆解）  
> **所属能力域**: C1-需求管理  
> **主要用户**: SE（系统工程师）  
> **页面类型**: 创建/编辑页

---

## 一、功能概述

### 1.1 功能定位
SSTS拆解是SE将Feature细化为系统需求（System）、技术需求（Technical）、安全需求（Safety）的核心功能，是需求工程的关键环节。

### 1.2 核心价值
- **需求细化**: 将Feature拆解为可验证的SSTS
- **分类管理**: 区分系统、技术、安全三类需求
- **安全合规**: 确保满足ISO 26262等安全标准
- **可追溯性**: 建立Feature到SSTS的追溯链

### 1.3 SSTS分类

| 类型 | 说明 | 示例 | ASIL等级 |
|------|------|------|---------|
| System | 系统需求 | 车道线检测系统需求 | 可选 |
| Technical | 技术需求 | 车道保持控制技术需求 | 可选 |
| Safety | 安全需求 | LKA安全需求 | 必填 |

## 三、表单字段

### 3.1 必填字段

| 字段名 | 字段说明 | 数据类型 | 验证规则 |
|--------|---------|---------|---------|
| SSTS类型 | 需求类型 | Enum | System/Technical/Safety |
| SSTS标题 | 需求标题 | String | 必填，5-100字符 |
| SSTS内容 | 详细描述 | Text | 必填，20-5000字符 |
| 需求分类 | 功能分类 | Enum | FUNCTIONAL/PERFORMANCE/SAFETY/INTERFACE |
| 负责人 | SSTS Owner | User | 必填，默认当前SE |
| 验证方法 | 验证手段 | Array | 至少选择一项 |

### 3.2 条件必填字段

| 字段名 | 条件 | 说明 |
|--------|------|------|
| ASIL等级 | type == 'Safety' | 安全需求必须指定ASIL等级 |
| 关联标准 | type == 'Safety' | 安全需求建议关联标准 |

### 3.3 选填字段

| 字段名 | 字段说明 | 数据类型 |
|--------|---------|---------|
| 验收标准 | AC列表 | Array<String> |
| 标签 | 分类标签 | Array<String> |
| 优先级 | 实现优先级 | Enum |

## 四、SSTS分类详解

### 4.1 System（系统需求）

**定义**: 描述系统应该做什么（What）

**示例**:
```
SSTS ID: ADAS-S001
标题: 车道线检测系统需求
内容: 系统应能够检测车道线，检测准确率≥95%，检测距离≥50m，
      支持直线、曲线、虚线、实线等多种车道线类型
分类: FUNCTIONAL
验证方法: 仿真测试、实车测试
```

### 4.2 Technical（技术需求）

**定义**: 描述系统如何实现（How）

**示例**:
```
SSTS ID: ADAS-S002
标题: 车道保持控制技术需求
内容: 系统应能够控制车辆保持在车道中央，横向偏差≤0.2m，
      采用PID控制算法，控制周期≤20ms
分类: PERFORMANCE
验证方法: 台架测试、实车测试
```

### 4.3 Safety（安全需求）

**定义**: 描述系统的安全约束（Safety Constraints）

**示例**:
```
SSTS ID: ADAS-S003
标题: LKA安全需求
内容: LKA功能应满足ASIL-D安全等级要求，包括故障检测、
      安全状态转换、驾驶员接管机制、故障降级策略
分类: SAFETY
ASIL等级: ASIL_D
验证方法: 仿真测试、台架测试、实车测试、形式化验证
关联标准: ISO 26262, GB/T 34590-2017
```
